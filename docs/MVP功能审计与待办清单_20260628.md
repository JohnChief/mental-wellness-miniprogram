## 一、项目现状概览

| 维度 | 实际实现 |
|------|---------|
| 前端 | 原生微信小程序（6 个页面） |
| 后端 | Flask + MySQL，部署微信云托管 `flask-ytaf` |

| 飞书机器人 | 长连接 + OpenAI 兼容 API |

### 已完成功能

- 首页、活动列表、活动详情、报名、我的页面、报名记录、协议页
- OpenID 登录 + 手动手机号报名

- 名额事务锁（`SELECT ... FOR UPDATE`）、防重复报名
- Admin 数据概览、活动 CRUD、上下架、报名筛选与签到
- Admin 用户搜索、排序、VIP 管理（单个+批量）
- Admin Session 登录、CSRF、二次确认、Toast 提示、审计日志
- 测试数据生成/恢复工具（受环境变量保护）
- 加载态、空数据态、错误态、满员态等基础状态覆盖

## 三、P0 — 上线前必须修复（影响数据正确性 / 合规）

| # | 问题 | 现状 | 影响 | 建议修复 |
|---|------|------|------|---------|
| P0-1 | **活动开始后仍可报名** | ❌ `registration_deadline=NULL` 时不校验 `event_time`，活动进行中也可提交报名 | 逻辑错误 | 后端在报名时加校验：`event_time` 已过 → 拒绝；前端隐藏报名按钮 |
| P0-2 | **注销后幽灵报名占名额** | ❌ 注销只匿名化用户字段，`registered`/`checked_in` 报名不改变状态 | 名额被无效用户占用 | 注销时自动取消未来活动中的 `registered` 状态报名，释放名额 |
| P0-3 | **Admin 无分页** | ❌ 用户/报名列表一次性查全表 | 数据量增大后响应慢、内存爆 | 服务端分页，默认 20-50 条/页，保留搜索/筛选参数 |
| P0-4 | **首页无活动时空状态** | ❌ `featured_event` 为 null 时首页仍渲染主视觉区域，可能空卡片 | 用户体验差 | 条件渲染：无活动时显示「暂无活动，敬请期待」 |
| P0-5 | **隐私版本号硬编码** | ❌ 前端写死 `2026-06-24`，更新需改代码发版；老用户不会被要求重新确认 | 合规风险 | 版本号从后端 settings 获取，登录/报名接口校验用户 `privacy_version` 是否为当前版本 |
| P0-6 | **活动名额可改到低于已报名数** | ❌ Admin 可把 `capacity` 从 20 改到 5，但已有 15 人已报名 | 数据矛盾 | 保存活动时校验 `capacity >= active_registration_count` |
| P0-7 | **环境变量缺失时不报错** | ❌ `SECRET_KEY` 等缺失时服务照常启动，运行时才暴露 | 排查成本高 | 启动时校验必需环境变量，缺失直接终止并输出明确错误 |

## 四、P1 — 体验与安全增强（建议 MVP 收尾前完成）

| # | 问题 | 现状 | 建议 |
|---|------|------|------|
| P1-1 | **Admin 登录无限速** | ❌ 无失败次数限制 | 加 Flask-Limiter 或手写限速，5 次失败锁定 15 分钟 |
| P1-2 | **Admin 并发编辑冲突** | ❌ 无版本号/更新时间校验，后提交覆盖先提交 | 活动编辑页加 `updated_at` 校验，冲突时提示 |
| P1-3 | **活动修改后无通知** | ❌ 时间/地点改了已报名用户完全不知道 | 本期可做：Admin 保存时提醒「已报名 X 人，是否需要通知」；实际通知渠道留二期 |
| P1-4 | **N+1 查询** | ❌ 活动列表每个活动额外 2 次查询统计报名数 | 用 GROUP BY 一次聚合查询各活动报名数 |
| P1-5 | **取消报名重复点击改写时间** | 🟡 多次取消会重写 `cancelled_at`，首次取消时间丢失 | 已取消状态幂等返回，不修改时间 |
| P1-6 | **名额为 0 时仍显示「立即报名」** | 🟡 前端未禁用按钮 | 检查剩余名额，=0 时按钮变灰 + 文案改为「名额已满」 |
| P1-7 | **报名截止无前端倒计时/状态** | 🟡 后端有校验，前端不展示 | 活动详情页显示截止倒计时，已过期隐藏按钮 |
| P1-8 | **寒启动冷启动** | 🟡 云托管长时间无请求会休眠，首访白屏 3-5s | 配最小实例数 ≥1 或加外部 /health 定时拨测 |
| P1-9 | **时区一致性** | 🟡 `datetime.now()` 无时区，依赖容器与 MySQL 一致 | 确认容器+MySQL 均为 `Asia/Shanghai`，或统一用 UTC 存储 |
| P1-10 | **图片加载失败无兜底** | 🟡 有 `cover_color` 兜底色但缺 `binderror` 占位图逻辑 | 前端 `<image>` 加 `binderror` 切默认占位图 |
