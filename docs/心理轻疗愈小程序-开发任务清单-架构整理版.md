# 心理轻疗愈小程序 - 开发任务清单（架构整理版）

> 状态说明（2026-06-25）：本文保留早期产品与架构规划作为背景资料。
> 当前实现已调整为 Flask + MySQL、微信 OpenID 登录、报名时手动填写手机号，
> 并已完成同仓 Admin 运营后台。实际状态以根目录 README、`backend/README.md`
> 和 `docs/数据库表结构.md` 为准。

版本：v0.3  
日期：2026-06-23  
阶段：测试号 / MVP 开发准备

## 1. 当前目标

本项目先完成一个可演示、可联调、可验收的微信小程序 MVP，核心验证：

- 用户进入小程序后可以浏览首页和活动信息
- 用户可以完成登录 / 注册 / 手机号授权
- 用户可以提交活动报名
- 管理员可以在后台查看活动、报名和用户信息
- 后续拿到正式主体后，可以迁移到正式小程序环境

本阶段优先保证主流程跑通，不提前实现复杂咨询、支付、内容系统、会员体系和精细化权限。

## 2. 已确认原则

- MVP 阶段以注册 / 报名流程为核心。
- 管理端采用轻量方案，不单独做复杂前端项目。
- 数据模型按实际需求逐步确定，但需要保留扩展空间。
- 敏感信息和合规不做重型系统，但从第一天保留基础底线。
- 测试号阶段如果无法使用微信云服务，可以先用本地后端和临时数据库完成联调。

## 3. 关键技术决策

### 3.1 前端路线

当前有两条可选路线，需要在正式开发前确定。

#### 方案 A：原生微信小程序

技术栈：

- WXML
- WXSS
- JavaScript
- 微信原生 API

适合情况：

- 项目长期只做微信小程序
- 希望减少框架层和编译链路
- 希望最贴近微信开发者工具和微信审核环境

优点：

- 调试链路最直接
- 与微信能力适配最好
- 项目复杂度低

缺点：

- 未来迁移 H5、App、其他小程序时成本较高
- 不能直接使用 Vue 生态和 uView Plus

#### 方案 B：uni-app Vue3

技术栈：

- uni-app
- Vue3
- uView Plus
- 编译到微信小程序

适合情况：

- 未来可能需要 H5、App、支付宝/抖音小程序等跨端能力
- 希望使用 Vue 组件化开发体验
- 希望使用 uView Plus 等成熟 UI 组件库

优点：

- 一套代码保留跨端可能
- Vue 组件化开发效率较高
- uView Plus 可减少基础 UI 开发量

缺点：

- 多一层编译框架
- 微信特殊能力仍需适配
- 调试和排错比原生小程序复杂

### 3.2 当前仓库状态

当前仓库是微信开发者工具生成的原生小程序模板，不是 uni-app 项目。

因此如果最终选择 uni-app，需要重新初始化前端项目骨架；如果选择原生小程序，则需要把任务清单中的 uni-app / uView Plus 内容改成原生小程序实现方式。

### 3.3 后端路线

MVP 阶段建议采用：

- FastAPI
- SQLAlchemy
- SQLite（仅本地开发 / 测试临时使用）
- Jinja2 + Pico.css 管理端

正式主体确认后，根据微信云服务可用性再决定：

#### 正式方案 A：FastAPI 继续保留

- FastAPI 部署到云托管或云服务器
- 数据库迁移到正式数据库
- 小程序继续通过 HTTPS API 调用后端

适合情况：

- 希望后端能力更独立
- 后续可能接入更多非微信渠道
- 业务逻辑较多，适合集中在 Python 后端

#### 正式方案 B：迁移到微信云开发

- 数据迁移到微信云数据库
- 部分接口改为云函数或云托管能力
- 小程序与微信生态集成更紧密

适合情况：

- 项目长期只服务微信小程序
- 希望减少服务器运维
- 业务逻辑相对简单

注意：如果从 FastAPI + SQLite 迁移到微信云数据库 / 云函数，不是简单替换数据库连接，需要提前把业务逻辑和数据访问隔离。

## 4. 推荐架构

### 4.1 整体结构

```text
小程序前端
  |
  | HTTPS API
  v
FastAPI 后端
  |
  | service 层
  v
repository 层
  |
  | 本地开发：SQLite
  | 正式阶段：微信云数据库 / 云托管数据库 / 其他正式数据库
  v
数据存储
```

### 4.2 后端分层

建议后端保持以下分层：

```text
api/router      对外接口
schemas         请求和响应结构
services        业务逻辑
repositories    数据访问
models          数据模型
admin           管理端页面
core            配置、认证、错误处理
```

这样本地 SQLite 和未来正式云服务可以通过 repository 层切换，报名、登录、活动上下架等业务流程不需要大改。

### 4.3 管理端原则

管理端可以与 FastAPI 放在同一个项目中：

- `/admin/login`
- `/admin/activities`
- `/admin/enrollments`
- `/admin/users`

管理端不直接操作数据库，应复用 service 层，避免小程序 API 和管理端产生两套业务规则。

## 6. MVP 数据模型

## 7. 合规和敏感信息底线

MVP 阶段不做复杂合规系统，但必须保留以下底线：

- 报名前必须勾选同意隐私政策和用户协议。
- 只收集完成报名所必需的信息。
- 不在前端长期缓存手机号、openid、报名内容。
- 后端日志不打印手机号、openid、报名表详情。
- 管理端必须登录后才能访问报名和用户信息。
- 隐私政策、用户协议、免责声明页面需要在小程序内可访问。

后续可以再补充：

- 账号注销
- 数据导出
- 手机号脱敏展示
- 角色权限
- 更完整的审计日志

## 8. 开发阶段任务

### 阶段 0：方向确认

- 确认前端选择：原生小程序或 uni-app Vue3
- 确认正式阶段后端方向：继续 FastAPI 或迁移微信云开发
- 确认品牌名称、Logo、主色
- 确认活动报名为主，还是报名 + 咨询并存
- 确认客服渠道

产出：

- 技术路线确认
- MVP 范围确认
- 页面清单确认

### 阶段 1：项目初始化

如果选择原生小程序：

- 清理微信默认模板页面
- 建立首页、活动、我的三个 tab
- 建立通用 API 请求封装
- 建立登录状态管理

如果选择 uni-app：

- 重新创建 uni-app Vue3 项目
- 配置 uView Plus
- 配置微信小程序编译和运行
- 建立页面和组件目录

后端：

- 创建 FastAPI 项目
- 配置 SQLAlchemy
- 配置 SQLite 本地数据库
- 配置基础错误处理
- 配置环境变量
- 创建管理端模板目录

产出：

- 可运行的小程序骨架
- 可运行的后端服务
- 可访问的管理端登录页

### 阶段 2：注册和登录流程

小程序：

- 调用微信登录
- 调用后端登录接口
- 保存 token

- 登录弹窗或登录页

后端：

- 微信 code 换 openid
- 创建或更新用户

- 手机号绑定接口

产出：

- 用户可以完成登录 / 注册 / 手机号绑定

### 阶段 3：活动和报名流程

小程序：

- 活动列表页
- 活动详情页

- 报名成功页

后端：

- 活动列表接口

- 提交报名接口
- 防重复报名

产出：

- 用户可以完整完成活动浏览和报名

### 阶段 4：管理端

管理端：

- 管理员登录
- 活动列表
- 活动新建 / 编辑 / 上下架

- 管理端鉴权

### 阶段 5：联调测试

测试路径：

- 未登录浏览首页
- 未登录浏览活动列表
- 点击报名后触发登录
- 登录后授权手机号

- 管理端查看报名

- token 过期

- 活动下架后访问

产出：

- MVP 主流程验收通过

- 正式上线准备清单

1. 先确认前端路线。
2. 如果选择 uni-app，重建前端项目骨架。
3. 如果选择原生小程序，清理当前模板并直接进入首页 / 活动 / 我的开发。

5. 优先实现登录、手机号授权、报名提交、管理端查看报名。

## 11. 架构审计结论

当前方案适合作为 MVP，但需要避免两个风险：

- 不要同时摇摆在原生小程序和 uni-app 之间。
- 不要把本地 SQLite 的临时方案写成不可替换的数据访问逻辑。

只要前端路线和后端长期方向先确定，本项目可以采用轻量架构推进；不需要一开始做复杂系统，但需要从第一天保留 service / repository 分层、基础登录鉴权、隐私协议勾选和管理端访问控制。
