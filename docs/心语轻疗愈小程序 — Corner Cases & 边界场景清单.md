## 🔴 专题：两人同时抢最后一个名额（超卖）

场景还原：活动名额 capacity=20，已报名 19 人，剩余 1 个名额。用户A和用户B几乎同时点击「立即报名」，两个请求几乎同时到达 Flask 后端。

**如果不用锁会怎样：**

请求A执行 SELECT COUNT\(*\) \.\.\. 返回 19，判断 19 \< 20 ✅，INSERT 报名*
*请求B执行 SELECT COUNT\(*\) \.\.\. 也返回 19（A还没提交），判断 19 \< 20 ✅，INSERT 报名
结果：报了21个人，名额超卖 ❌

**MySQL 行级锁方案（推荐）：**

步骤1：START TRANSACTION
步骤2：SELECT capacity FROM events WHERE id = :event\_id FOR UPDATE（锁住这一行）
步骤3：SELECT COUNT\(\*\) FROM registrations WHERE event\_id=X AND status IN \('registered','checked\_in'\)
步骤4：IF 报名数 \>= capacity THEN ROLLBACK; RETURN "名额已满"
步骤5：INSERT INTO registrations \(\.\.\.\) VALUES \(\.\.\.\)
步骤6：COMMIT（释放锁）

**为什么 SELECT FOR UPDATE 就够了：**

- 锁在 events 表的对应行上，而不是 registrations 表

- 同一活动的并发报名请求会在步骤2排队串行执行

- 不同活动之间互不影响（行锁，不是表锁）

- 配合 UNIQUE\(event\_id, user\_id\) 约束做二次兜底

**需要验证的事：**

- ☐ MySQL 引擎是 InnoDB（MyISAM 不支持行锁）

- ☐ events 表 id 有索引（主键自带）

- ☐ 名额统计用的是 registered \+ checked\_in，不含 cancelled（文档已确认）

- ☐ 并发测试：用 ab 或 wrk 模拟 10 个并发请求抢最后 1 个名额，确认只有 1 个成功

乐观锁方案不推荐：秒杀场景下冲突率高，大量请求需要重试，用户体验差。行级悲观锁更合适。

## 11\. 本次审计新增场景

### 11.1 🔴 用户注销后遗留报名继续占用名额

- 风险：注销逻辑会匿名化报名资料，但不会改变 `registered` / `checked_in` 状态。未开始活动中的“幽灵报名”会继续占名额；用户重新注册后还能以新 `user_id` 再报名一次。

- 状态：❌ 未覆盖

- 建议：注销前要求用户处理未结束报名，或自动取消未来活动中状态为 `registered` 的记录；已签到和历史活动保留匿名化记录。需要补事务测试。

### 11.2 🔴 公网域名与 X-WX-OPENID 信任边界

- 风险：后端直接信任 `X-WX-OPENID`。需要确认微信云托管公网入口是否会删除/覆盖客户端自行提交的同名请求头；否则直接访问公网域名可能伪造身份。

- 状态：⚙️ 平台安全验证

- 建议：用公网域名发送伪造 `X-WX-OPENID` 做黑盒测试；必要时将用户 API 限制为微信私有协议入口，或验证云托管注入的额外可信标识。`ALLOW_DEV_OPENID` 在生产必须为 `false`。

### 11.3 ✅ 旧 Admin JSON API 已移除

- 原风险：Session/CSRF Web Admin 和 `X-ADMIN-KEY` JSON API 分别维护业务规则，容易出现权限、校验和审计分叉。

- 状态：✅ 已处理

- 处理：旧 `/admin/api/*` 路由、`X-ADMIN-KEY` 鉴权和 `ADMIN_API_KEY` 配置已删除，只保留 Session + CSRF Web Admin。测试明确断言旧接口返回 404。

### 11.4 关键环境变量缺失时未快速失败

- 风险：`SECRET_KEY`、管理员密码或 MySQL 配置缺失时，服务可能启动后才在用户请求中暴露错误，增加排障成本。

- 状态：❌ 未覆盖

- 建议：生产启动时校验必需环境变量；缺失时直接终止启动并输出不含敏感值的明确错误。

### 11.5 活动名额被改小到低于当前报名数

- 风险：Admin 可将 `capacity` 调整到小于已报名/已签到人数，造成“容量 10、实际 15 人”的矛盾状态。

- 状态：❌ 未覆盖

- 建议：保存活动时校验 `capacity >= active_registration_count`；若确需缩容，应先人工处理报名并记录审计原因。

### 11.6 Admin 列表没有分页

- 风险：用户和报名数据增长后会一次查询并渲染全部记录，导致响应慢、内存上升和页面难以操作。

- 状态：❌ 未覆盖

- 建议：用户、报名、审计列表增加服务端分页，默认每页 20-50 条，并保留搜索、筛选和排序参数。

### 11.7 活动列表剩余名额存在 N+1 查询

- 风险：每序列化一个活动会额外执行两次报名统计查询；活动数量增加后数据库查询数线性放大。

- 状态：❌ 未优化

- 建议：使用分组聚合一次查询各活动有效报名数，或维护可靠的计数列。

### 11.8 时间与时区一致性

- 风险：代码使用无时区 `datetime.now()`，依赖容器和数据库时区一致。时区配置漂移会影响截止时间、活动排序和审计时间。

- 状态：🟡 依赖部署配置

- 建议：确认容器与 MySQL 都为 `Asia/Shanghai`；后续统一使用 UTC 存储、展示时转换。

### 11.9 首页没有在线活动时的空状态

- 风险：`featured_event` 可能为 `null`，但首页仍渲染主视觉区域并读取其字段，可能出现空卡片或异常体验。

- 状态：❌ 未覆盖

- 建议：首页根据 `featuredEvent` 条件渲染；没有活动时显示“暂无活动”并隐藏主推入口。

### 11.10 隐私政策版本号硬编码

- 风险：小程序页面使用固定 `2026-06-24`，更新隐私政策后必须改代码发版，且老用户不会被自动要求重新确认。

- 状态：❌ 未覆盖

- 建议：将当前隐私版本放入 `settings` 或专用配置，由后端返回；登录和报名接口校验提交版本是否为当前版本。

### 11.11 测试数据影响真实名额与运营统计

- 风险：开启测试工具后生成的报名与真实报名使用同一张表，会占用名额并进入仪表盘统计。

- 状态：🟡 已隔离入口，未隔离数据

- 建议：测试环境优先使用独立数据库/云环境；若必须共库，增加 `is_test` 字段并从正式统计和名额计算中排除。生产必须关闭测试工具。

### 11.12 多次点击取消报名会重写取消时间

- 风险：取消接口对已取消记录仍返回成功并重写 `cancelled_at`，导致首次取消时间丢失。

- 状态：🟡 功能可用，审计语义不严谨

- 建议：已取消时直接幂等返回原记录，不修改时间；或记录独立状态变更历史。
