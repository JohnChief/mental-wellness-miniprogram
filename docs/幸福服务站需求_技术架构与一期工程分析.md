# 幸福服务站小程序需求：技术架构与一期工程分析

> 依据：甲方《幸福服务站小程序需求文档（补充版）》  
> 分析重点：技术架构可落地性、一期开工范围、当前工程改造差距

## 一、需求性质判断

甲方需求不是普通展示型小程序，而是“适老化 C 端小程序 + 多角色 B 端运营后台 + 热线/视频/工单/客户档案/设备联动”的业务闭环系统。

核心业务闭环为：

1. 老人/居民通过小程序发起热线、预约、门店服务或设备呼叫。
2. 后台生成客户档案、通话记录、服务预约或工单。
3. 接听员、咨询师、运营管理员按权限处理、转派、记录、归档。
4. 后台沉淀公益客户、付费客户、高危心理来访者、机器人呼叫等数据。
5. 后续再接 AI 心理倾诉、心理测评、数字人、独立 App、社区接口。

因此，一期技术方案应按“业务中台型系统”设计，而不是简单堆页面。

## 二、建议总体技术架构

### 2.1 前端层

建议保留当前原生微信小程序路线，原因是：

- 当前工程已经是原生微信小程序。
- 需求大量依赖微信生态能力：授权登录、分享、保存图片、位置导航、客服/通话入口、云托管调用。
- 适老化 UI 需要细控字体、按钮、表单、无障碍状态，原生实现可控性较高。

一期 C 端建议页面收敛为：

- 首页：热线入口、服务入口、门店入口、机器人绑定入口。
- 服务大厅：心理陪伴、家政便民、养老关怀、门店休闲四类服务。
- 统一预约表单：不同服务类型共用一套预约底座。
- 热线页：语音/视频呼叫入口、通话记录。
- 我的中心：个人档案、我的工单/订单、设备管理、电子热线卡。

### 2.2 后端层

当前工程为 Flask + MySQL + 微信云托管，可继续作为一期底座，但需要从“活动报名系统”升级为“服务工单系统”。

建议后端拆成以下业务模块：

- 用户与来访者档案模块：微信 OpenID、基础信息、紧急联系人、隐私授权、客户标签。
- 服务项目模块：心理咨询、家政、养老关怀、门店预约等统一配置。
- 预约/订单模块：C 端提交预约，后台受理、确认、取消、完成。
- 热线通话模块：记录语音/视频呼叫流水、关联客户和工单。
- 工单模块：待接单、处理中、线下预约、服务完成、归档。
- 角色权限模块：热线接听员、心理咨询师、运营管理员。
- 设备模块：机器人绑定、解绑、测试呼叫、呼叫记录同步。
- 统计模块：公益客户数、待处理工单、预约量、机器人呼叫量、付费转化。
- 审计日志模块：客户档案查看、修改、导出、工单处理全程留痕。

### 2.3 数据层

一期数据模型建议至少新增：

| 领域 | 建议表 | 说明 |
|---|---|---|
| 用户/档案 | `visitors` / `visitor_profiles` | 老人、居民、公益客户、付费客户等统一档案 |
| 联系人 | `emergency_contacts` | 紧急联系人、亲属信息 |
| 标签 | `visitor_tags` / `visitor_tag_links` | 独居老人、高危心理、公益客户等 |
| 服务 | `service_categories` / `service_items` | 四大服务分类与具体服务项目 |
| 预约 | `appointments` | 统一预约单，覆盖心理、家政、门店等 |
| 工单 | `work_orders` / `work_order_logs` | 工单主表和流转记录 |
| 通话 | `call_records` | 热线语音/视频呼叫流水 |
| 话术 | `script_templates` | 接待、安抚、危机干预话术 |
| 门店 | `stores` / `store_staff` | 社区服务站点与在岗人员 |
| 设备 | `devices` / `device_call_records` | 老人机器人绑定与呼叫记录 |
| 权限 | `admin_users` / `roles` / `role_permissions` | 多后台角色与数据权限 |
| 审计 | `admin_audits` 扩展 | 当前已有审计表，可扩展动作和对象类型 |

敏感字段如心理痛点、咨询备注、紧急联系人、通话摘要，应设计字段级加密或至少应用层加密，并限制后台列表默认展示完整内容。

### 2.4 第三方/平台依赖

一期需要尽早确认的外部能力：

| 能力 | 关键问题 | 建议 |
|---|---|---|
| 语音热线 | 是否直接拨打手机号，还是系统内记录通话 | 一期可先用拨号入口 + 后台手动/半自动补录通话记录 |
| 视频通话 | 微信小程序视频能力、RTC 服务、客服通话能力选型 | 不建议一期开自研实时音视频，先明确资质、成本、可用 SDK |
| 通话录音/录像留存 | 文档要求本地 + 云端双备份、至少 2 年 | 一期必须先定供应商和合规存储方案，否则只能做“通话记录元数据” |
| 机器人设备 | 是否已有设备厂商 API | 一期先做设备绑定、测试呼叫、模拟回调接口，真实联调作为里程碑 |
| 社区数据接口 | 社区是否有接口规范 | 一期只预留导入/同步接口，避免卡住主流程 |
| 数据备份 | 隐私数据、通话记录、工单留存 | MySQL 自动备份 + 对象存储备份 + 操作日志 |

## 三、当前工程适配分析

当前仓库已具备：

- 原生微信小程序基础结构。
- Flask + MySQL 后端。
- 微信云托管调用模式。
- OpenID 用户识别。
- 管理后台 Session 登录、CSRF、登录限速、审计日志。
- 活动、报名、用户、设置等基础 CRUD 经验。

当前主要差距：

- 业务模型仍是“活动/报名”，与甲方新需求的“服务/预约/工单/档案”不匹配。
- 后台目前是单管理员思路，缺少多角色、多权限、数据隔离。
- 没有客户档案、咨询记录、紧急联系人、服务工单、热线记录、设备绑定模型。
- 没有通话/视频/机器人/社区接口的真实集成方案。
- 隐私与心理咨询合规要求明显高于现有活动报名 MVP。

结论：现有工程可以作为一期技术底座，但不能只在原活动模块上改文案；应做一次业务域重构。

## 四、一期工程范围建议

甲方文档写明一期为：

> 上线热线、预约、工单、客户档案、机器人基础联动。

建议把一期拆成“必须上线”和“预留/弱实现”两层。

### 4.1 一期必须上线

1. C 端首页与适老化基础 UI
   - 大字体、大按钮、高对比度。
   - 一键拨打热线。
   - 服务大厅入口。
   - 门店入口。
   - 我的中心。

2. 统一预约能力
   - 心理咨询预约。
   - 家政/助餐预约。
   - 上门探视/养老关怀预约。
   - 门店到店预约。
   - 统一预约状态：待处理、处理中、已完成、已取消。

3. 客户档案
   - 基础信息。
   - 紧急联系人。
   - 客户标签。
   - 历史预约/工单。
   - 隐私授权记录。

4. 后台工单管理
   - 待处理工单列表。
   - 工单详情。
   - 分配咨询师/服务人员。
   - 状态流转。
   - 跟进记录。
   - 归档。

5. 后台多角色登录
   - 运营管理员。
   - 热线接听员。
   - 心理咨询师。
   - 一期可以先做页面级权限和数据范围控制，不做过度复杂 RBAC。

6. 门店与服务项目管理
   - 门店信息维护。
   - 服务分类和服务项目维护。
   - 预约单关联服务与门店。

7. 机器人基础联动
   - 设备绑定/解绑。
   - 设备状态。
   - 测试呼叫。
   - 机器人呼叫记录生成工单。
   - 若厂商接口未确定，可先做标准回调接口和模拟联调。

8. 基础统计看板
   - 今日预约。
   - 待处理工单。
   - 客户数量。
   - 热线记录数量。
   - 机器人呼叫数量。

### 4.2 一期建议弱实现/预留

以下内容不建议一期完整实现，否则工期和风险会明显上升：

- 实时视频通话完整系统：先做入口和记录，真实 RTC/录制能力待供应商确认后单独排期。
- AI 心理倾诉机器人：一期只预留接口，不上线实际对话。
- 线上心理测评量表：放二期。
- 数字人：放三期。
- 独立 App：放三期。
- 复杂营销自动化：一期只做电子热线卡和基础分享。
- 社区系统实时同步：一期做导入或接口预留。

## 五、一期里程碑建议

### 阶段 0：需求澄清与技术定版

交付物：

- 一期功能边界确认表。
- 角色权限矩阵。
- 工单状态流转图。
- 数据字段清单。
- 视频通话/机器人供应商接口确认。

必须确认的问题：

- 视频通话是否必须一期真实可用。
- 通话记录是否包含录音/录像文件，还是只记录通话元数据。
- 机器人厂商是否已有 API 文档。
- 心理咨询记录的可见范围和留存要求。
- 高危来访者紧急联系人通知方式：电话、短信、人工提醒还是系统通知。

### 阶段 1：数据模型与后台底座

重点：

- 新增客户档案、服务项目、预约、工单、门店、设备、角色权限表。
- 改造后台登录为多账号/多角色。
- 扩展审计日志。

### 阶段 2：C 端核心流程

重点：

- 首页适老化改版。
- 服务大厅与统一预约表单。
- 我的预约/工单记录。
- 个人档案与紧急联系人。
- 设备绑定页。

### 阶段 3：B 端运营流程

重点：

- 后台仪表盘。
- 客户档案管理。
- 工单列表和详情。
- 工单分配、状态流转、跟进记录。
- 服务项目和门店配置。
- 话术库基础维护。

### 阶段 4：热线与机器人联动

重点：

- 热线拨号入口。
- 通话记录录入/同步。
- 机器人呼叫回调。
- 呼叫自动生成工单。
- 设备测试呼叫。

### 阶段 5：联调、合规与上线

重点：

- 权限隔离测试。
- 敏感数据脱敏与审计。
- 数据备份。
- 弱网/重复提交/老人误触测试。
- 真机适老化测试。

## 六、主要技术风险

| 风险 | 影响 | 建议 |
|---|---|---|
| 视频通话能力不清 | 可能拖慢一期上线 | 一期先拆成“拨号入口 + 通话记录”，视频 RTC 单独确认 |
| 通话记录双备份要求高 | 涉及存储、成本、合规 | 尽早确认是否录音/录像，选择对象存储和生命周期策略 |
| 心理咨询隐私数据敏感 | 合规风险高 | 权限隔离、字段加密、审计日志、导出限制必须一期做 |
| 机器人接口不确定 | 联调风险高 | 先定义标准回调协议，厂商未到位时用模拟器 |
| 需求范围过大 | 一期延期风险高 | 一期聚焦热线入口、预约、工单、档案、设备基础联动 |
| 老年用户误操作 | 工单脏数据、体验差 | 大按钮、少输入、二次确认、语音输入作为增强项 |
| 后台多角色复杂度 | 容易过度设计 | 一期先页面级权限 + 数据范围，二期再细颗粒 RBAC |

## 七、一期技术结论

一期建议定位为“幸福服务站业务闭环 MVP”，目标不是把甲方所有长期规划一次做完，而是先跑通：

老人/居民发起需求 → 后台生成预约/工单 → 人员处理和记录 → 客户档案沉淀 → 基础统计和设备呼叫联动。

推荐技术路线：

- 小程序：继续使用原生微信小程序。
- 后端：继续使用 Flask + MySQL + 微信云托管。
- 后台：短期可继续 Jinja2 后台，若后续模块快速变多，二期再拆 Vue/React 管理端。
- 存储：MySQL 存结构化数据，对象存储存图片、资质、通话录音/录像等文件。
- 权限：一期做多账号、多角色、页面权限、核心数据审计。
- AI/数字人/测评：只预留接口，不进入一期主交付。

当前工程可复用技术底座，但需要重建业务域模型。建议不要在现有“活动报名”模型上强行套新需求，而是把活动报名能力抽象升级为“服务项目 + 预约 + 工单”的通用流程。
